좋아. 지금 1차 실험을 **full-body generation 없이 trajectory planning loop만으로 goal 달성률을 보는 것**으로 잡는 건 합리적이야. 다만 그러려면 “window sample”이 아니라 “episode-level simulation unit”으로 다시 정리해야 함.

**1. 현재 stage1 v2 sampling 기준**

현재 만든 v2 sampling은 대략 이 기준이었어.

```text
window length = 102 frames
past = 30
future = 72
stride = 8
ego = move phase only
others = move/action 상관없이 포함 가능
max others = 3
total characters = ego + up to 3 others
max samples per ego = 32
character count priority = 4 chars > 3 chars > 2 chars
others all-window-move ratio target >= 75%
```

즉 이건 **Stage1 prediction 학습용 window sampling**임.

그런데 지금 하려는 실험은:

```text
initial world state
+ ego action queue
+ fixed others trajectories / occupancies
→ closed-loop trajectory planning
→ ego가 queue goal들을 얼마나 달성하는가
```

라서 기존 window sample만으로는 부족함. 이제는 **test episode sample**을 따로 만들어야 해.

**2. 새 test episode 생성 기준**

기본적으로는 이렇게 잡는 게 맞아.

```text
scene split = test
episode length = 충분히 김
ego has action queue
others trajectories are fixed from data
ego starts before first goal
```

필수 조건:

```text
1. ego action queue가 존재해야 함
2. 각 ego action에 body_goal / goal area가 있어야 함
3. ego initial position이 scene valid region에 있어야 함
4. goal area가 local crop에 들어오는 문제가 아니라, global scene 기준으로 정의되어야 함
5. others는 최대 3명
6. others slot ordering deterministic
```

그리고 conflict filtering을 추가.

**3. 추가해야 할 conflict 기준**

네 말대로 others끼리 conflict를 먼저 제거해야 함.

```text
others-others dynamic conflict:
  for all t, for all i != j:
    ||p_i(t) - p_j(t)|| >= 0.5m
```

이건 strict하게 하면 데이터가 많이 줄 수 있으니, 처음엔 violation ratio 기준도 가능함.

```text
allowed violation ratio < 1~2%
```

하지만 test benchmark를 깨끗하게 만들려면 처음엔 strict 추천.

그 다음, ego action goal과 others의 persistent occupancy 충돌 제거.

```text
for each ego action goal area G:
  for each other j:
    other final action goal area not overlap G
    other final-frame root disk radius 0.25m not overlap G
```

여기서 네가 말한 “0.5m 지름 원”이면 root disk는:

```text
radius = 0.25m
```

그리고 이것도 추가하는 게 좋음.

```text
ego initial root disk가 any other initial/future persistent occupancy와 겹치지 않음
ego action goal끼리도 너무 가까운 중복 goal 제거
goal area가 완전히 obstacle 내부인 이상한 case는 따로 flag
```

다만 goal이 clearance 밖에 있는 건 무조건 제거하면 안 됨. 우리가 corridor handling을 만든 이유가 있으니까, 제거하지 말고 metadata로 남기는 게 맞음.

```text
goal_safe / goal_unsafe flag 저장
```

**4. Episode loop 설계**

trajectory-only loop는 이렇게 가면 됨.

```text
state:
  ego current root
  ego current velocity
  current phase = MOVE / WAIT / ACTION
  action queue
  others future trajectory source
  time index
```

매 window:

```text
1. observe current world state
2. Stage1 predictor로 ego/others future prediction
3. optimizer로 ego root trajectory 30 frames planning
4. Phase/Queue manager update
5. execute root trajectory 일부 frames
6. update time/current state
```

full-body action generation 대신 Action phase는 data에서 action trajectory를 가져오거나 단순 occupancy로 처리.

**5. Action phase를 어떻게 대체할지**

1차 실험에서는 action motion 자체가 중요하지 않으니, 이렇게 간소화하면 됨.

```text
MOVE:
  optimizer trajectory execute

WAIT:
  near-stationary trajectory execute

ACTION:
  ego root를 goal area 내부/근처에 고정
  action duration만큼 time consume
  action complete 조건 만족 시 pop queue
```

만약 “training data에서 action trajectory를 따온다”면 더 자연스럽지만, goal 달성률 실험에서는 꼭 필요하진 않음. 오히려 root만 보면:

```text
action = stay at goal for required duration
```

이 더 명확함.

**6. 중요한 문제: optimizer가 goal까지 도달하나?**

현재 optimizer horizon은:

```text
30 frames = 1 sec
```

그래서 한 번의 planning window로 goal 도착을 강제하지 않는 게 맞음. 대신 closed-loop에서 매 window마다 goal distance가 줄어야 함.

필요한 건 “terminal goal force”보다 **progress term**이야.

```text
J_goal =
  mean distance_to_goal_area(p_t)
+ terminal_weight * distance_to_goal_area(p_T)
```

그리고 Phase Manager가 도착 판정:

```text
if distance_to_goal_area(current_pos) < arrival_threshold:
    switch MOVE -> ACTION
```

여기서는 `goal_tolerance`가 다시 필요해짐. optimizer cost에는 안 썼지만, loop state transition에는 필요함.

추천:

```text
arrival_threshold = 0.25~0.35m
required_hold_frames = maybe 5~10
```

즉:

```text
optimizer:
  goal로 진행하게 함

phase manager:
  goal 도착 여부를 판정함
```

이렇게 분리해야 함.

**7. 지금 의논해야 할 핵심 결정들**

우리가 정해야 할 것:

```text
1. test episode length
2. action queue 길이
3. action duration을 실제 데이터에서 쓸지, fixed duration으로 둘지
4. MOVE window execute length
   - optimize horizon은 30 frames
   - 실제 execute는 10/15/30 중 선택

5. arrival threshold
6. wait timeout
7. strict sequence vs local reordering
8. others trajectory를 GT로 고정할지, Stage1 prediction으로 rolling할지
```

1차 실험 추천은 가장 단순하게:

```text
others = GT future fixed
ego planning = Stage1 prediction + optimizer
execute length = 15 frames
horizon = 30 frames
strict queue
no reordering
WAIT allowed
ACTION = fixed root hold at goal
arrival_threshold = 0.3m
action duration = dataset segment duration or capped fixed duration
```
